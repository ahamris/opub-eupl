## ⚠️ PROBLEMEN EN VERBETERPUNTEN

### 1. **File Header (regels 1-15)**
**Probleem:** De file begint met een informele intro tekst die er niet thuishoort:
```
Nice, this is the fun part 😄
Below is a **complete new `oo_llm.txt`** focused on:
...
```

**Oplossing:** Verwijder deze intro, begin direct met de titel en doel.

### 2. **Incorrecte Code Block Syntax**
**Probleem:** Regel 16 en 204 gebruiken `````txt` in plaats van ```json voor JSON schemas.

**Oplossing:** Verander naar ```json voor alle JSON schema blocks.

### 3. **Detail Schema - `creatiedatum` Requirement**
**Probleem:** Regel 284 zegt `required: ["pid", "creatiedatum"]`, maar:
- Niet alle documenten hebben `creatiedatum` (sommige hebben alleen `openbaarmakingsdatum`)
- In de CTER documenten zien we dat sommige documenten `creatiedatum` hebben, andere niet

**Oplossing:** Maak `creatiedatum` optioneel:
```json
"required": ["pid"]
```

### 4. **Search Hit Schema - Te Strikte Requirements**
**Probleem:** Regel 137-144 vereist veel velden die mogelijk niet altijd aanwezig zijn:
- `weblocatie` kan null zijn
- `aanbieder` kan ontbreken in sommige responses

**Oplossing:** Maak meer velden optioneel, alleen `id` en `pid` als required:
```json
"required": ["id", "pid"]
```

### 5. **Ontbrekende Informatie**
**Probleem:** De guide mist:
- Informatie over de werkelijke response structuur (is het een array of object?)
- Voorbeelden van echte API responses
- Informatie over error responses

**Oplossing:** Voeg toe:
- Sectie met voorbeeld responses
- Error handling patterns
- Link naar bestaande implementatie

### 6. **Inconsistentie met Laravel Implementatie**
**Probleem:** De guide is framework-agnostic, maar:
- Jullie hebben al een Laravel implementatie
- De guide zou kunnen verwijzen naar de bestaande DTOs en Services

**Oplossing:** Voeg sectie toe die linkt naar:
- `App\DataTransferObjects\OpenOverheid\OpenOverheidSearchQuery`
- `App\Services\OpenOverheid\OpenOverheidSearchService`
- `App\Models\OpenOverheidDocument`

### 7. **Detail Document - `versies` Structuur**
**Probleem:** In de CTER documenten zien we dat `versies` op top-level staat, niet in `document`. De guide heeft dit correct, maar de beschrijving op regel 269 is onduidelijk.

**Oplossing:** Maak duidelijker dat `versies` een top-level array is.

### 8. **Bestand Structuur in `versies`**
**Probleem:** Regel 547 zegt `required: ["id", "url", "mime-type"]`, maar:
- In sommige documenten kan `url` ontbreken (sommige hebben alleen `id`)
- `url` is niet altijd aanwezig in search results

**Oplossing:** Maak `url` optioneel:
```json
"required": ["id", "mime-type"]
```

### 4. **ID Handling Duidelijker**
De guide legt uit dat `id` vaak `_1` suffix heeft, maar dit is niet altijd consistent. Maak dit duidelijker:
```markdown
**ID Handling:**
- Search results: `document.id` = "oep-..._1" (with version suffix)
- Detail endpoint: Use base ID without suffix
- Extraction: Either strip `_1` or extract from `pid` URL path
- Example: "oep-abc123_1" → use "oep-abc123" for detail call
```
